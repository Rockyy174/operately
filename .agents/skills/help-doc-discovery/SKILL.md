# Help Doc Discovery

You are **agent 1** in a two-agent documentation pipeline:

1. **Discovery (this skill)** — parse code changes, infer user-visible behavior,
   and produce a structured backlog in terse technical language.
2. **Writing (downstream agent)** — turn each backlog item into end-user help
   content using the website repo's writing guidelines.

Do **not** draft help pages, MDX, sidebar entries, or polished user-facing copy.
Your job ends at a complete, accurate discovery brief another agent can write from.

For git commands, code-tracing maps, classification rules, and output templates,
read [reference.md](reference.md).

## Scope

**In scope:**

- Git history, diffs, PRs, tags, and release ranges in the **operately** repo
- User-visible behavior inferred from frontend code, feature tests, and UI copy
- Mapping discoveries to existing help pages in the sibling **website** repo
  (when available at `../website/`)

**Out of scope:**

- Writing or editing help MDX
- Website sidebar, frontmatter, screenshots, or Starlight formatting
- API reference docs under `help/api/` (auto-generated by CI)
- Developer docs, architecture notes, or contribution guides

## Workflow

### 1. Define the change window

Confirm what to analyze before reading code:

- **Commits** — SHA range, branch tip, or merged PR
- **Time** — `since` date or between tags (e.g. last release → `main`)
- **Paths** — default to user-facing surfaces; narrow only when asked

Record the scope in the output header (range, commit count, paths included).

### 2. Collect changes from git

Start with git, not memory:

```bash
git log --oneline <range> -- app/assets/js/ app/test/features/ turboui/src/
git diff --stat <base>..<head> -- app/assets/js/ turboui/src/
```

Group related commits (same feature, PR, or area). Read commit messages for
intent, but **verify every claim in the diff** — messages lie or omit UI impact.

See [reference.md — Git analysis](reference.md#git-analysis) for more commands.

### 3. Filter for documentation impact

For each change group, decide whether end users need new or updated help docs.

**Usually document:**

- New screens, flows, modals, settings, or navigation paths
- Changed steps, labels, permissions, or limits users see in the UI
- New billing, access-control, or onboarding behavior
- Removed or relocated actions users previously relied on

**Usually skip** (note as `no_doc_needed` with reason):

- Backend-only refactors with no UI diff
- Test, CI, lint, or build changes
- Internal code moves with identical rendered UI
- Auto-generated API reference (handled separately)

When unsure, include the item with `confidence: low` rather than dropping it.

### 4. Trace code to user-visible behavior

For each item that passes the filter, read the diff and follow imports until you
can describe what a user **sees and does** — not how Elixir or GraphQL works.

**Primary surfaces:**

| Surface | Path | Use for |
| ------- | ---- | ------- |
| Pages | `app/assets/js/pages/` | Routes, layout, navigation |
| Features | `app/assets/js/features/` | Modals, forms, feature UI |
| Shared UI | `turboui/src/` | Billing, tables, banners, pickers |
| Feature tests | `app/test/features/` | End-to-end flows and labels |
| Page tests | `app/assets/js/**/*.spec.ts` | Component labels and behavior |

Extract exact UI strings (button labels, menu items, dialog titles, field
labels, toast text). Record navigation paths and permission gates visible in
the frontend.

Use `app/lib/` and operations only to **find which page renders a feature**,
then return to frontend source. Do not document backend internals in the brief.

See [reference.md — Code tracing](reference.md#code-tracing) for the full map.

### 5. Cross-check existing documentation

Search `../website/src/content/docs/help/` for pages covering the same feature
or task. Record:

- Existing slug(s) that likely need an **update**
- Gaps where a **new** page is probably needed
- `none` when no related page exists

Match by task and feature area, not by file name alone.

### 6. Emit the backlog

Write one brief per documentation unit (usually one user task or one concept).
Use the template in [reference.md — Output format](reference.md#output-format).

Keep language **terse and technical**:

- Good: "Billing page gates render on `can_manage_billing`; non-managers see read-only plan summary."
- Bad: "Learn how to manage your billing settings with our easy-to-use interface."

Include `open_questions` when the diff alone cannot confirm behavior (e.g.
feature flag default, edge case only covered in backend).

### 7. Review before handoff

- [ ] Scope header matches the git range analyzed
- [ ] Every item has commits, key files, and inferred user-visible behavior
- [ ] UI labels are quoted exactly from source, not paraphrased
- [ ] Each item has `action`, `area`, `confidence`, and existing-doc mapping
- [ ] No MDX, sidebar edits, or user-facing prose drafted
- [ ] Skipped changes are listed in `no_doc_needed` with reasons

## Definition of Done

A discovery task is complete when:

- The backlog covers all user-facing changes in scope (or explains exclusions)
- A writing agent could produce accurate help pages from the briefs alone
- Technical details are sufficient; marketing tone is absent
